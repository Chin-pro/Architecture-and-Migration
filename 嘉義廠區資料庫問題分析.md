# 你第一步不是改 code，而是查 `getMapName()` 被誰用

你在 IDE 裡先做：

```
Find Usages / Call Hierarchy
```

或直接搜尋：

```
getMapName(
vendorName
vendor_name
Map<
put(
get(
```

你要分清楚 `getMapName()` 現在到底被用在哪些地方。

大概會有兩種情況。

---

## 情況 A：`getMapName()` 只用在 vendor lookup

例如：

```java
map.put(getMapName(vendor), vendor);
Vendor vendor = map.get(getMapName(request));
```

如果它只負責 vendor mapping，那可以考慮改。

---

## 情況 B：`getMapName()` 也被其他地方用

例如：

```java
getMapName() 用在畫面顯示
getMapName() 用在報表名稱
getMapName() 用在 log 顯示
getMapName() 用在其他模組的 key
getMapName() 用在文具以外的功能
```

那就**不要直接改原本的 `getMapName()`**。

你應該新增一個更明確的方法，例如：

```java
private String getVendorIdentityKey(String siteId, String vendorCode) {
    return siteId + "|" + vendorCode;
}
```

或：

```java
private String getVendorIdentityKey(String siteId, String vendorName) {
    return siteId + "|" + vendorName;
}
```

這樣可以避免舊功能被你誤傷。

---

# 我會建議你優先採用「新增方法」，不要直接改舊方法

因為 `getMapName()` 這個名字太模糊。

它可能原本意思是：

```
取得顯示名稱
```

也可能是：

```
取得 Map 的 key
```

也可能是：

```
取得 vendor name 對應值
```

名字不清楚的方法，最好不要直接改它的語意。

比較安全的做法是新增：

```java
private String buildVendorKey(String siteId, String vendorCode) {
    return siteId + "|" + vendorCode;
}
```

如果沒有 `vendorCode`，才用：

```java
private String buildVendorKey(String siteId, String vendorName) {
    return siteId + "|" + vendorName;
}
```

但注意，如果兩個參數都是 `String, String`，Java 不能靠參數名稱 overload，所以實務上可以寫成：

```java
private String buildVendorKeyByCode(String siteId, String vendorCode) {
    return siteId + "|" + vendorCode;
}

private String buildVendorKeyByName(String siteId, String vendorName) {
    return siteId + "|" + vendorName;
}
```

---

# 第二步：確認上傳與刪除流程用的是同一套 vendor mapping 嗎？

你要找這兩條流程：

```
上傳文具項目
刪除文具項目
```

然後問自己：

```
它們是不是都會透過 getMapName() 找 vendor？
它們是不是都只用 vendor_name？
它們有沒有 site_id？
它們有沒有 vendor_code？
```

如果上傳和刪除都壞，很可能共用問題在這裡：

```
vendor lookup
vendor permission check
vendor item ownership check
Map key 產生邏輯
```

你不要只修上傳。

因為這次同時有「上傳」和「刪除」問題，所以要抓共用根源。

---

# 第三步：建立修正前的 baseline

在你改之前，先用你公司內部環境確認這幾種案例。

至少測：

```
1. 台南原本 vendor 上傳是否正常
2. 台南原本 vendor 刪除是否正常
3. 嘉義新 vendor 上傳是否失敗
4. 嘉義新 vendor 刪除是否失敗
5. 嘉義與台南是否存在相同 vendor_name
```

你不一定要寫正式測試，至少要有手動測試紀錄。

如果公司可以用工具，這裡可以用：

```
Postman / Apifox：手動打 API
REST Assured：寫 API regression test
SQL log：確認實際 where 條件
IDE Find Usages：確認方法影響範圍
Git diff：確認自己到底改了哪些地方
```

你之前說希望遇到這類問題時先提醒工具，這次就是典型可以用工具降低風險的情境。

---

# 第四步：先不要大改，只改 vendor lookup 的 identity

理想修正方向是：

```
從 vendor_name
改成 site_id + vendor_code
```

如果沒有 vendor_code，就退到：

```
site_id + vendor_name
```

例如原本可能是：

```java
Map<String, Vendor> vendorMap = new HashMap<>();

for (Vendor vendor : vendors) {
    vendorMap.put(vendor.getVendorName(), vendor);
}
```

你不要直接變成亂改一堆地方。

先改成：

```java
Map<String, Vendor> vendorMap = new HashMap<>();

for (Vendor vendor : vendors) {
    String key = buildVendorKeyByCode(vendor.getSiteId(), vendor.getVendorCode());
    vendorMap.put(key, vendor);
}
```

取資料時也要同樣規則：

```java
String key = buildVendorKeyByCode(siteId, vendorCode);
Vendor vendor = vendorMap.get(key);
```

重點是：

```
put 的 key 規則 = get 的 key 規則
```

這句很重要。

---

# 第五步：如果 request 沒有 site_id，你不能硬修

如果上傳或刪除 request 裡沒有 `site_id`，那你要找 site_id 從哪裡來。

可能來源有：

```
登入者資訊
session
token
vendor account 對應資料
前端目前選擇的 site
URL parameter
既有資料表關聯
```

如果後端完全拿不到 `site_id`，那你就無法可靠區分：

```
台南 ABC 文具
嘉義 ABC 文具
```

這時候你要回報：

```
目前後端輸入資訊不足，若只傳 vendor_name，無法區分跨廠區同名 vendor。需要上傳/刪除流程補 site_id 或 vendor_code。
```

這就是 PO 說「修改 template」可能想解的問題。

---

# 第六步：加防呆，不要讓錯誤靜默發生

如果查 vendor 時查不到，你不要讓它默默變成 null 或拿錯第一筆。

不要這樣：

```java
Vendor vendor = vendorMap.get(key);
```

然後後面直接用。

比較好：

```java
Vendor vendor = vendorMap.get(key);

if (vendor == null) {
    throw new IllegalArgumentException(
        "Vendor not found for siteId=" + siteId + ", vendorCode=" + vendorCode
    );
}
```

如果公司不適合 throw `IllegalArgumentException`，也要回傳明確錯誤。

重點是不要讓系統：

```
找不到嘉義 vendor → 自動拿台南 vendor
```

這種錯誤最危險。

---

# 第七步：測試順序

你修完後，不要只測嘉義。

要照這個順序測：

```
1. 嘉義同名 vendor 可以上傳
2. 嘉義同名 vendor 可以刪除
3. 台南同名 vendor 仍然可以上傳
4. 台南同名 vendor 仍然可以刪除
5. 台南和嘉義不會互相刪到對方資料
6. 原本沒有同名問題的 vendor 仍然正常
7. 其他使用 getMapName() 的功能沒有壞
```

這裡第 7 點最重要，因為你擔心的就是這個。

---

# 你的安全修正策略應該是這樣

我會建議你照這個版本做：

```
1. 不直接改 getMapName()
2. 先查 getMapName() 所有呼叫點
3. 如果它只用在文具 vendor lookup，才考慮改
4. 如果它被多處共用，新增 buildVendorKeyByCode / buildVendorKeyByName
5. 只在上傳與刪除流程替換 key 產生邏輯
6. 保留舊方法，避免影響其他功能
7. 加 log 驗證 site_id + vendor_code 是否正確
8. 測嘉義、台南、同名、非同名案例
```

---

# 你可以跟主管或 PO 這樣說

```
我會先確認 getMapName() 的使用範圍，避免直接修改共用方法造成其他功能 regression。若目前上傳與刪除流程是使用 vendor_name 作為 mapping key，我會優先將這兩個流程的 vendor lookup 調整為 site_id + vendor_code；若 vendor_code 不可用，則至少改成 site_id + vendor_name。

同時我會確認 request / template 是否有提供 site_id 或 vendor_code。如果目前輸入資料只有 vendor_name，那後端無法穩定區分嘉義與台南同名 vendor，可能需要同步調整輸入格式。
```

---

# 最重要的一句話

你現在不要做：

```
看到 getMapName 可疑 → 直接改 getMapName
```

你要做：

```
先查 getMapName 影響範圍 → 確認上傳/刪除共用點 → 新增明確的 vendor identity key → 小範圍替換 → 回歸測試
```

你的擔心是成熟工程師該有的擔心。

真正安全的修法不是「把可疑方法改掉」，而是**先控制變更範圍**。
